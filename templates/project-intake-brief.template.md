# Project Intake Brief

Esta plantilla ayuda a entregar suficiente contexto inicial al Project Orchestrator sin convertir el onboarding en un formulario pesado.

Puedes completar solo lo que conozcas o adjuntar un documento que contenga esta información. Los campos sin evidencia deben quedar como `Desconocido`; no es necesario inventarlos para comenzar.

## Versión mínima

Para iniciar, procura cubrir al menos:

```text
Nombre del proyecto:
Producto objetivo: [nuevo / existente]
Propósito:
Usuario principal:
Problema que resuelve:
Alcance inicial:
Fuentes disponibles:
```

`Producto objetivo` se refiere al producto que se quiere construir o continuar. La existencia de un sistema anterior, una migración o una referencia histórica no convierte automáticamente al producto objetivo en existente.

## Versión recomendada

```text
Nombre del proyecto:

Producto objetivo:
- Estado: [nuevo / existente]
- Tipo de iniciativa: [producto nuevo / evolución / reconstrucción / migración selectiva / otro]

Propósito:

Usuario o cliente principal:

Problema que resuelve:

Resultado esperado:

Alcance inicial:

Fuera de alcance:

Activos que ya existen:
- código;
- repositorios;
- documentación;
- usuarios;
- infraestructura;
- integraciones;
- diseños;
- datos.

Sistemas anteriores o relacionados:

Relación con esos sistemas:
- referencia;
- reemplazo;
- migración selectiva;
- integración;
- continuidad;
- ninguna.

Restricciones conocidas:
- tiempo;
- coste;
- seguridad;
- cumplimiento;
- tecnología;
- equipo.

Etapa o prioridad actual:

Decisiones confirmadas:

Supuestos:

Preguntas abiertas:

Fuentes adjuntas o enlazadas:
```

## Cómo debe usarla el orquestador

El Project Orchestrator debe:

1. leer primero todas las fuentes entregadas;
2. extraer de ellas la información equivalente a este brief;
3. distinguir hechos, preferencias, supuestos, propuestas y decisiones;
4. no volver a preguntar algo que ya esté respondido por una fuente;
5. marcar como `Desconocido` lo que no pueda verificar;
6. preguntar progresivamente solo por los vacíos que bloqueen el siguiente paso;
7. producir una síntesis estructurada antes de preparar desarrollo.

Un PDF, documento, wiki o repositorio puede reemplazar esta plantilla cuando entregue contexto suficiente.
